1. A cosa serve davvero THERADIOMICS?

THERADIOMICS non è solo una dashboard che calcola AUC. È uno strumento di study design radiomico.

Serve a capire se un dataset è plausibile per sviluppare una firma predittiva binaria compatta, stabile e statisticamente difendibile.

Frase chiave: non basta ottenere una bella AUC. Bisogna capire se quella AUC è credibile, riproducibile e difendibile.

2. Il flusso generale

La dashboard prende le feature radiomiche, le aggrega a livello paziente, elimina feature ridondanti, seleziona candidate feature, valuta modelli compatti, fa test di stabilità, simula la numerosità campionaria e aggiunge guardrail analitici.

Dataset → patient-level aggregation → pruning → LASSO
      → Nested CV / LOOCV / permutation / bootstrap
      → candidate models
      → Monte Carlo sample size
      → analytical guardrails
      → pruning threshold sweep
Spiegazione felina: prima togliamo i gomitoli uguali, poi vediamo quale gomitolo predice meglio la ciotola piena, poi controlliamo se il gatto ci arriva davvero o se è stato solo fortunato.

3. Lesioni e pazienti non sono la stessa cosa

Nel file iniziale puoi avere molte lesioni, per esempio 204. Questo però non significa avere 204 osservazioni statisticamente indipendenti.

Se un paziente ha più lesioni, quelle lesioni condividono biologia, terapia, risposta clinica, imaging, rumore sistematico e criteri di etichettatura.

Regola d’oro: per addestrare e validare modelli predittivi, il vero N è il numero di pazienti indipendenti, non il numero di lesioni.

4. Outcome, evento e prevalenza

La dashboard lavora con un outcome binario:

0 = non evento / non responder
1 = evento / responder

La prevalenza è la quota di pazienti con outcome positivo.

prevalenza = numero eventi / numero totale pazienti

Esempio: 20 responder / 35 pazienti = 0.571 = 57.1%.

Un modello binario non ha bisogno solo di pazienti. Ha bisogno di abbastanza esempi in entrambe le classi.

5. Feature radiomiche

Una feature radiomica è un numero che descrive un aspetto quantitativo dell’immagine: forma, intensità, texture, eterogeneità, distribuzione dei voxel.

La radiomica genera molte feature, ma molte possono essere duplicate, correlate, instabili o rumorose.

La domanda non è solo: “qual è la feature migliore?”. La domanda giusta è: “questa feature rimane importante se cambio leggermente il dataset?”.

6. Correlation pruning

Il pruning elimina feature troppo correlate tra loro. Se due feature dicono quasi la stessa cosa, tenerle entrambe aumenta il rischio di overfitting.

SogliaEffettoRischio
0.60Pruning molto aggressivoPuò buttare via segnale utile
0.75Forte riduzioneUtile per test conservativi
0.85Compromesso ragionevoleDefault pratico
0.90Più feature disponibiliPiù rischio ridondanza
0.95Molte feature rimangonoPiù rischio overfitting
Se la performance cambia drasticamente al cambiare della soglia di pruning, il risultato è fragile.

7. LASSO in parole semplici

LASSO è una regressione logistica con penalizzazione. Ogni feature che entra nel modello “costa”. Se una feature non serve davvero, il suo coefficiente viene spinto a zero.

La selezione LASSO fatta sul dataset completo è descrittiva. La performance vera va stimata con la selezione rifatta dentro la cross-validation.

8. Modello finale compatto

In radiomica small-sample, il modello finale deve essere piccolo.

Numero feature finaliUso consigliato
1Modello primario più difendibile
2–3Modello compatto esplorativo
5Limite alto, da trattare con prudenza
10Stress test / probabile overfitting

9. AUC: utile, ma non basta

L’AUC misura quanto bene il modello ordina i pazienti. Se prendo un responder e un non-responder a caso, l’AUC è la probabilità che il modello assegni rischio più alto al responder.

AUCInterpretazione
0.50Come tirare una moneta
0.60Debole
0.70Potenzialmente utile
0.80Buona
0.90Eccellente, ma sospetta se N è piccolo

10. Nested Cross-Validation

La Nested CV è una validazione interna severa. In ogni fold separa training e test, seleziona le feature solo sul training, addestra il modello e predice il test.

Il test fold non deve mai partecipare alla selezione delle feature. Questo evita leakage.

11. LOOCV

LOOCV significa Leave-One-Out Cross-Validation: ogni volta si lascia fuori un paziente, si addestra sugli altri e si predice il paziente escluso.

È utile con dataset piccoli, ma può essere variabile. Va interpretata come confronto, non come verità assoluta.

12. Sensibilità e specificità

Sensibilità
Tra i veri eventi, quanti vengono riconosciuti?
Specificità
Tra i veri non-eventi, quanti vengono riconosciuti?

Sensibilità e specificità dipendono dalla soglia di classificazione, spesso 0.5. L’AUC invece valuta l’ordinamento globale.

13. Permutation test

Il permutation test mescola a caso le etichette e verifica se il modello riesce comunque a ottenere AUC simili.

Se anche le etichette randomizzate danno spesso AUC alte, il risultato osservato non è convincente.

Se durante una permutazione LASSO seleziona zero feature, la pipeline usa un modello solo-intercetta: quella permutazione non contiene segnale utile.

14. Bootstrap stability

Il bootstrap ripete molte volte la selezione delle feature su campioni casuali con reinserimento. Se una feature compare spesso, è più stabile.

Frequenza bootstrapInterpretazione
<20%Instabile
20–50%Debole/moderata
50–75%Interessante
>75%Molto stabile

15. Candidate models

La dashboard confronta modelli candidati a 1, 3 e 5 feature, spesso derivate dalle feature più stabili al bootstrap.

Il modello più interessante non è sempre quello con AUC massima. Un modello piccolo, stabile e spiegabile può essere più pubblicabile di un modello più grande con AUC leggermente superiore.

16. Monte Carlo sample-size simulation

La simulazione Monte Carlo crea tanti dataset sintetici e chiede: se il vero segnale avesse una certa AUC attesa, con N pazienti quanto spesso riuscirei a osservarlo?

N = 150
power = 0.82

Questo significa che, nelle simulazioni, con 150 pazienti l’82% degli esperimenti raggiungeva la soglia di successo definita.

17. Guardrail analitici

I guardrail analitici non sostituiscono il Monte Carlo. Lo affiancano.

Monte Carlo
Con questo N, quanto spesso ottengo una AUC utile?
Guardrail
Questo N è difendibile rispetto a prevalenza, eventi, predittori e overfitting?

N event-rate

Numero minimo di pazienti necessario per stimare la prevalenza con un certo margine di errore.

MargineInterpretazione
±5%Molto rigoroso
±7.5%Intermedio
±10%Realistico per feasibility

N shrinkage

Numero minimo di pazienti necessario per controllare l’overfitting dato un certo numero di predittori.

18. k_max

k_max indica quanti predittori finali posso permettermi dato un certo N.

N = 150
k_max ≈ 2

Significa che con 150 pazienti, sotto quelle ipotesi, il modello dovrebbe avere al massimo circa 2 predittori.

Se il modello candidato ha 5 feature ma k_max è 2, la performance può essere interessante, ma il modello è troppo grande per quel N.

19. Sweep delle soglie di pruning

Lo sweep ripete l’analisi per più soglie di pruning.

Dalla dashboard

  • Soglia pruning singola: un solo run.
  • Sweep soglie pruning: più run consecutivi.
  • Feature finali k: consigliato 1, massimo prudente 3.
  • Simulazioni Monte Carlo: 100 rapido, 500 buono, 1000 più stabile.

21. Come interpretare un risultato

Risultato interessante

  • Nested CV AUC ≥ 0.70.
  • LOOCV non contraddice completamente.
  • Permutation test convincente.
  • Bootstrap stability buona.
  • Modello a 1–3 feature.
  • k_max compatibile con il numero di predittori.
  • Sweep non instabile.

Risultato sospetto

  • AUC alta solo con 10 feature.
  • AUC alta solo a una soglia di pruning.
  • Feature instabili al bootstrap.
  • Permutation test non significativo.
  • k_max molto inferiore alle feature usate.

22. Checklist finale

  • Quanti pazienti veri ho?
  • Quanti eventi e non-eventi ho?
  • Il modello usa 1–3 feature oppure troppe?
  • La feature principale è stabile al bootstrap?
  • La AUC regge in Nested CV?
  • Il permutation test è convincente?
  • Lo sweep pruning dà risultati coerenti?
  • k_max consente davvero quel numero di predittori?
  • La stima Monte Carlo è compatibile con i guardrail?
  • Riesco a spiegare il modello clinicamente?
Se la risposta è sì alla maggior parte di queste domande, il modello è più difendibile.
